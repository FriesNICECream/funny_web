# Research: Stitch Media Data 仪表盘

## 决策 1: 复用现有 React + Vite 单页结构承接新页面

- **Decision**: 在现有 `src/App.jsx` 的已登录分支中复用仪表盘入口，通过新增 `src/features/media-data/` 特性目录承接页面内容，而不是把 Stitch 导出的整页 HTML 直接塞入现有组件树。
- **Rationale**: 当前仓库已经具备认证页、仪表盘页和通用 UI 目录，继续沿用单页结构可以减少路由与会话层改动，并符合“页面层负责组合、复杂逻辑下沉到 feature”的宪章要求。
- **Alternatives considered**:
  - 直接将 Stitch HTML 作为静态页面嵌入：落地最快，但会绕过现有 React 结构，后续状态管理、主题、复用和响应式修补成本过高。
  - 完全重写 `App.jsx` 为多路由结构：扩展性更强，但超出本期聚焦的“已登录仪表盘改版”范围。

## 决策 2: 以 Stitch 信息结构为蓝本，转为本地可维护的 MOCK 视图模型

- **Decision**: 将 Stitch 页面中的标题区、导航项、资源卡片、资源链接、筛选入口整理为显式视图模型，并先由本地 MOCK 数据驱动前端组件，而不是让组件直接依赖原始 HTML 文本块或未定义接口。
- **Rationale**: 该页面明显包含重复卡片结构、导航激活态和响应式展示规则，先建立统一视图模型有助于后续接入真实数据，也便于空状态、缺省字段和无结果状态处理。
- **Alternatives considered**:
  - 只保留静态 JSX 常量：前期简单，但无法稳定支撑筛选、排序、搜索与状态反馈。
  - 直接绑定未来未知接口：会把当前任务阻塞在后端契约上，不适合本期前端改版。

## 决策 3: 将媒体数据页做成独立 dashboard feature，而不是继续堆叠到现有 DashboardPage

- **Decision**: 保留 `src/features/dashboard/` 作为壳层，新增媒体数据页面专属的容器、卡片、导航和数据文件，避免让现有 `DashboardPage.jsx` 持续膨胀。
- **Rationale**: 当前 `DashboardPage.jsx` 已承担布局、侧边栏和首屏内容组合；若直接在单文件中加入媒体数据页的所有逻辑，会继续违背宪章中对页面层清晰边界的约束。
- **Alternatives considered**:
  - 直接在 `DashboardPage.jsx` 内联所有新布局：修改面最小，但后续难以维护。
  - 新建完全平级的应用入口：结构更纯粹，但对现有登录后流程改动过大。

## 决策 4: 响应式策略以“桌面常驻侧栏 + 移动端折叠入口”为主

- **Decision**: 桌面端保留侧边导航、头部搜索和多列卡片布局；移动端将侧边导航折叠为可显式展开的入口，卡片网格降为单列或双列，并优先保证链接区域可触达。
- **Rationale**: Stitch 页面本身就包含明显的桌面信息密度和移动收拢需求，当前仓库侧边栏组件也已有移动端开关能力，延续这一模式最稳妥。
- **Alternatives considered**:
  - 移动端完全隐藏部分导航和筛选能力：实现简单，但无法满足同流程可用性要求。
  - 桌面与移动维护两套不同页面：灵活性高，但维护成本翻倍。

## 决策 5: 新增或替换基础交互时按 shadcn/ui 兼容方式推进

- **Decision**: 本特性实现中优先复用现有 `src/components/ui/` 里的按钮、输入框、侧边栏等基础组件；如发现缺失，再补足 `components.json` 与必要的 shadcn/ui 初始化边界，并保持新旧组件入口兼容。
- **Rationale**: 项目宪章明确要求新增或替换基础 UI 组件时优先使用 shadcn/ui 体系，同时仓库当前尚未存在 `components.json`。因此计划中需要把“补齐初始化边界或兼容策略”作为显式基础任务，而不是在实现中临时决定。
- **Alternatives considered**:
  - 继续自由手写新的基础按钮、卡片、输入样式：短期快，但会进一步扩大散装 UI。
  - 立即大规模迁移整个项目到完整 shadcn/ui：方向正确，但超出本期范围。
